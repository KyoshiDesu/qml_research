# Web Application Architecture, Model Integration, and Evaluation

## 1. Purpose and scope

The SVM vs QSVM Model Lab turns the trained artifacts under `output_gpu/` into
an interactive research application. It supports:

- prediction from one manually entered observation;
- batch prediction from a CSV containing new observations;
- comparison of classical SVM and quantum-kernel SVM predictions;
- evaluation when ground-truth labels are supplied;
- export of predictions and Markdown/JSON evaluation reports; and
- display of the historical held-out metrics generated by the training
  experiment.

The application is a research and demonstration system. It is not a clinical
decision system, a calibrated probability service, or a production
high-throughput inference API.

## 2. Architectural rationale

### 2.1 Why a web application

A web interface is appropriate because the main users are expected to inspect
research models rather than integrate a prediction API into another service.
The workflow requires forms, editable tabular input, CSV upload, metric
visualization, and report downloads. A browser-based application makes these
functions accessible without requiring users to write Python.

### 2.2 Why Streamlit

Streamlit was selected because it provides the required research-dashboard
features directly in Python:

- numeric and text input widgets;
- CSV upload and download;
- tables and metric cards;
- session state;
- caching of expensive model loads; and
- a test harness for executing an application without opening a browser.

This avoids maintaining a separate JavaScript frontend and HTTP API while the
application is still being used as a model evaluation tool. A frontend/API
split would add deployment and schema-management cost without currently adding
research value.

The business logic is deliberately kept outside the Streamlit file. The
inference and reporting modules can therefore be reused later by a FastAPI
service, command-line tool, scheduled evaluation job, or notebook.

### 2.3 Architectural boundaries

```text
User / CSV
    |
    v
Streamlit presentation layer
streamlit_app.py
    |
    +--------------------------+
    |                          |
    v                          v
Inference service          Reporting service
inference.py               app_reporting.py
    |                          |
    v                          v
Trusted Joblib bundles     Metrics + Markdown/JSON
output_gpu/                report payloads
    |
    +------------+------------------+
    |                               |
    v                               v
Classical SVM path              QSVM compatibility path
scikit-learn Pipeline           quantum_compat.py
                                PennyLane + precomputed SVC
```

The boundaries have the following responsibilities:

| Layer | Responsibility | Does not own |
|---|---|---|
| Presentation | Widgets, user feedback, tables, downloads, model selection | Model preprocessing or metric calculations |
| Inference | Artifact discovery, loading, input contract, preprocessing, prediction, label decoding | Streamlit rendering |
| Quantum compatibility | Reconstructing the serialized PennyLane kernel classifier and selecting a simulation device | Dataset/UI logic |
| Reporting | Application metrics, confusion matrices, report payloads and Markdown | Model execution |
| Headless runner | Reusing inference and reporting without a browser | Duplicate inference logic |

This separation is the main maintainability decision in the architecture. The
UI and the headless test command exercise the same prediction and report code,
which reduces the risk that a report script behaves differently from the web
application.

## 4. Model artifact contract

The application discovers files matching:

```text
output_gpu/outputs_<dataset>_gpu/models/<kind>_final_model.joblib
```

Four dataset schemas and two model kinds are currently available:

| Dataset | Raw features | Classes | Models |
|---|---:|---:|---|
| Iris | 4 | 3 | SVM, QSVM |
| Wine | 13 | 3 | SVM, QSVM |
| Breast cancer | 30 | 2 | SVM, QSVM |
| Heart disease | 13 | 2 | SVM, QSVM |

The classical artifact contains the fitted preprocessor, fitted scikit-learn
pipeline, label encoder, and selected parameters. The QSVM artifact contains:

- the fitted raw-feature preprocessor;
- the fitted PCA or passthrough feature reducer;
- the fitted `[0, π]` quantum feature scaler;
- the fitted quantum-kernel classifier and its saved training inputs;
- the label encoder;
- selected quantum-feature-map parameters; and
- quantum backend metadata.

Keeping fitted preprocessing objects inside each model bundle is essential.
Application data is transformed with the same imputation, scaling, encoding,
and dimensionality-reduction parameters used at training time. Refitting any
of these transformations in the application would cause training/serving skew.

## 5. Model implementation in the application

### 5.1 Common input contract

`validate_input()` in `src/qsvm_vm_compare/inference.py` obtains the expected
raw feature names from `preprocessor.feature_names_in_`. It then:

1. rejects missing required columns;
2. orders columns exactly as the fitted preprocessor expects;
3. ignores unrelated columns such as a supplied target; and
4. rejects non-numeric values in numeric feature columns.

The manual form uses the fitted imputer statistics as defaults. Consequently,
the application does not maintain a second hard-coded feature schema that
could drift away from the model.

### 5.2 Classical SVM inference

```text
raw DataFrame
    -> saved preprocessing pipeline
       -> median imputation
       -> standardization
    -> saved SVC
    -> encoded class
    -> saved LabelEncoder
    -> user-facing label
```

The application passes the validated raw DataFrame to the saved fitted
pipeline. This is the shortest and safest serving path because scikit-learn
owns the complete preprocessing-to-classifier transformation.

### 5.3 Quantum-kernel SVM inference

```text
raw DataFrame
    -> saved preprocessor
    -> saved PCA / SelectKBest reducer
    -> saved MinMaxScaler to [0, pi]
    -> quantum feature map
    -> K(new rows, saved training rows)
    -> SVC(kernel="precomputed")
    -> encoded class
    -> saved LabelEncoder
    -> user-facing label
```

For datasets with more than four processed features, the saved PCA projects
the data into four dimensions. The scaler maps these features to quantum
rotation angles. The PennyLane feature map then evaluates a fidelity-like
kernel between each new row and the observations retained by the fitted
classifier. The resulting kernel matrix is passed to the fitted classical
`SVC(kernel="precomputed")`.

The application exposes two execution devices:

- `default.qubit`, a portable CPU simulator; and
- `lightning.gpu`, the GPU device used by the recorded training experiments.

Training on a quantum simulator does not make inference independent of the
simulator. Each new row still requires kernel-circuit evaluation against the
saved training rows, which explains why QSVM inference is slower than
classical SVM inference.

### 5.4 Serialized-class compatibility

The trained QSVM artifacts were serialized while the standalone experiment
script was running as `__main__`. Such a pickle records the class path as
`__main__.PennyLaneKernelSVC`, which is not normally available when Streamlit
loads it.

`src/qsvm_vm_compare/quantum_compat.py` provides an application-owned
implementation of this class. `load_model_bundle()` registers it under the
expected `__main__` name immediately before loading a trusted QSVM artifact.
The compatibility class restores the saved SVC and training inputs while
rebuilding non-serializable PennyLane kernel functions lazily.

This bridge is unusual but necessary for the existing artifacts. It makes the
models usable without rerunning expensive training. A future artifact format
should serialize a class from a stable package module so that this alias is no
longer required.

## 6. Code walkthrough

### 6.1 `streamlit_app.py`

- `cached_bundle()` caches model loading because Joblib deserialization should
  not occur on every Streamlit rerun.
- `prediction_controls()` limits model choices to artifacts actually
  discovered for the selected dataset and exposes the QSVM execution device.
- The single-prediction tab builds fields from the fitted feature schema and
  displays the decoded label and elapsed time.
- The batch tab supplies a schema-correct CSV template, accepts new rows,
  optionally identifies a ground-truth column, runs either or both models, and
  calculates model agreement.
- Results are retained in Streamlit session state and can be downloaded as
  CSV, Markdown, or JSON.
- The documentation tab separates historical held-out experiment performance
  from metrics calculated on the currently uploaded data.

### 6.2 `src/qsvm_vm_compare/inference.py`

- `ModelSpec` is the immutable identifier used by discovery and Streamlit
  caching.
- `discover_models()` derives dataset and model kind from the output layout,
  so adding another correctly structured artifact does not require changing a
  registry in the UI.
- `model_metadata()` reads training metrics, selected parameters, dataset
  summary, and backend information without coupling inference to report files.
- `load_model_bundle()` enforces required bundle keys and installs the legacy
  QSVM class alias.
- `default_input_row()` derives manual-form defaults from fitted imputer
  statistics.
- `validate_input()` is the serving schema boundary.
- `predict()` implements the two model paths and returns both encoded and
  decoded predictions plus measured elapsed time.

### 6.3 `src/qsvm_vm_compare/quantum_compat.py`

This module reconstructs the PennyLane feature maps used by the training
workflow:

- Z, ZZ, and configurable Pauli feature maps;
- linear, circular, and full entanglement;
- inverse feature-map application;
- the all-zero-state projector used for kernel evaluation; and
- kernel matrix construction against saved training rows.

The `use_device()` method invalidates the lazily created kernel when switching
between CPU and GPU simulation.

### 6.4 `src/qsvm_vm_compare/app_reporting.py`

The report layer calculates accuracy, balanced accuracy, macro F1, a
classification report, and a confusion matrix when ground truth is available.
It preserves historical training-test metrics separately from application
metrics and produces both a structured JSON payload and readable Markdown.

### 6.5 `scripts/test_application.py`

This is a headless adapter over the same inference and reporting services. It
provides a reproducible application-level smoke test and creates:

- `predictions.csv`;
- `report.md`; and
- `report.json`.

It does not contain an alternative model implementation.

## 7. What is distinctive about this architecture

1. **Artifact-driven UI:** feature fields and available models are derived from
   fitted artifacts rather than duplicated configuration.
2. **One inference implementation:** browser and command-line evaluation share
   the same service functions.
3. **Comparable model paths:** classical and quantum models accept the same raw
   DataFrame contract even though their internal transformations differ.
4. **Historical/live metric separation:** the app avoids presenting a small
   uploaded batch as if it were the training experiment's held-out test.
5. **Portable QSVM execution:** GPU-trained artifacts can be evaluated using a
   CPU simulator, subject to latency.
6. **Legacy artifact recovery:** the compatibility module makes existing
   `__main__`-serialized research models deployable without retraining.
7. **Report as a first-class result:** predictions, provenance fields,
   evaluation metrics, confusion matrices, and limitations are available as
   machine-readable JSON as well as Markdown.

## 11. Required fixes and recommended improvements

### 11.1 High priority

1. **Remove mutable device switching from shared cached models.** Streamlit
   caches one mutable bundle and `use_device()` changes that model in place.
   Concurrent sessions choosing different QSVM devices could race. Cache a
   separate model per `(artifact, device)` or create a request-local predictor.
2. **Clear or key result state by dataset and input.** `last_predictions` and
   `last_report` are global session keys. Changing the dataset after a run can
   leave results from the previous dataset visible until another prediction is
   executed.
3. **Add QSVM request limits.** Set maximum CSV rows, execution time, and
   preferably an asynchronous job boundary. Quantum kernel cost grows with
   uploaded rows multiplied by saved training rows and can block the Streamlit
   process.
4. **Strengthen target validation.** Reject missing target values, report
   unknown labels explicitly, and prevent empty uploads from reaching
   scikit-learn metrics.
5. **Add artifact integrity and compatibility checks.** Store and verify an
   artifact SHA-256, schema version, package versions, training commit, and
   model type before loading.
6. **Replace `__main__` serialization.** Move the training-time
   `PennyLaneKernelSVC` into the package and regenerate artifacts with a stable
   module path. The current compatibility alias should be treated as a
   migration mechanism.

### 11.2 Test coverage improvements

1. Add integration tests that load every real SVM artifact and validate an
   expected prediction.
2. Add a marked, slower QSVM test using `default.qubit` to verify a complete
   kernel prediction in continuous integration.
3. Add Streamlit interaction tests for manual prediction, CSV upload, invalid
   schema feedback, target selection, metrics, and downloads.
4. Add edge cases for empty files, duplicate columns, NaN targets, infinity,
   malformed numeric values, unknown categorical values, and unsupported model
   kinds.
5. Add report schema tests covering confusion matrices, classification-report
   serialization, and multiple simultaneous models.
6. Add performance regression thresholds for classical inference and a
   separately configured QSVM benchmark.

### 11.3 Model and evaluation improvements

1. Evaluate on a genuinely independent external dataset. The constructed
   three-row sample proves application correctness, not statistical
   generalization.
2. Add feature ranges, units, and out-of-distribution warnings to the model
   schema. Correct column names alone do not guarantee meaningful inputs.
3. Add repeated or larger external evaluation with confidence intervals.
4. If confidence scores are required, calibrate them on data not used to fit
   the classifier. The current application correctly avoids presenting SVM
   decision values as probabilities.
5. Monitor live feature drift, class distribution, latency, failures, and
   observed performance when delayed labels become available.

### 11.4 Production-readiness improvements

1. Add authentication, authorization, upload-size limits, privacy rules, and
   audit logging before accepting sensitive datasets.
2. Run model inference behind an API/worker layer if concurrent users or
   long-running QSVM jobs are expected.
3. Include artifact hash, application version, dependency versions, request
   identifier, and input-data hash in every exported report.
4. Containerize the application separately from the training environment. The
   portable CPU application should not require CUDA packages.
5. Never accept user-uploaded Joblib files. Joblib uses pickle semantics and
   can execute arbitrary code during loading; only versioned, trusted local
   artifacts should be loaded.

## 12. Overall assessment

The architecture is appropriate for an interactive research application. It
has clean separation between presentation, inference, quantum compatibility,
and reporting; it reuses fitted preprocessing correctly; and it provides a
headless path for reproducible application-level evaluation.

The implemented checks demonstrate that the classical path, QSVM artifact
loading, CPU quantum prediction, initial web rendering, and report generation
work. The next engineering priorities are eliminating shared mutable QSVM
state, preventing stale cross-dataset results, placing limits around expensive
quantum jobs, and expanding the test suite from component and smoke tests to
full UI and real-artifact integration coverage.
